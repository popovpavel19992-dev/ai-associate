## Goal

Plaintiff-side, pre-litigation AI demand-letter drafting integrated into the existing 3.4 Settlement demand-letter tracker. Lawyer clicks "Draft with AI" on a case's demand-letters section, AI classifies the claim type, RAG injects facts and statutes, Claude drafts five editable sections, lawyer exports PDF using the existing renderer, then continues through the existing draft → sent → responded lifecycle.

## Claim types (4)

`contract` | `personal_injury` | `employment` | `debt`. Classifier returns one of these four with confidence 0–1. ≥0.7 → auto-pick. <0.7 → lawyer picks from ranked dropdown. Stored in new column `claim_type` on `case_demand_letters`.

`claim_type` (legal theory) is **orthogonal** to the existing `letter_type` enum (`initial_demand|pre_litigation|pre_trial|response_to_demand`, which is lifecycle stage). Both columns coexist.

### Database (migration 0059)

```sql
-- Extend existing case_demand_letters table
ALTER TABLE case_demand_letters
  ADD COLUMN claim_type text
    CHECK (claim_type IS NULL OR claim_type IN ('contract','personal_injury','employment','debt')),
  ADD COLUMN claim_type_confidence numeric(3,2),
  ADD COLUMN cache_hash text,
  ADD COLUMN ai_summary text,
  ADD COLUMN ai_generated boolean NOT NULL DEFAULT false;

CREATE UNIQUE INDEX case_demand_letters_org_cache_hash_uq
  ON case_demand_letters(org_id, cache_hash)
  WHERE cache_hash IS NOT NULL;

-- New table for AI-generated sections
CREATE TABLE case_demand_letter_sections (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  letter_id uuid NOT NULL
    REFERENCES case_demand_letters(id) ON DELETE CASCADE,
  section_key text NOT NULL
    CHECK (section_key IN ('header','facts','legal_basis','demand','consequences')),
  content_md text NOT NULL,
  regenerated_at timestamptz,
  UNIQUE (letter_id, section_key)
);
CREATE INDEX case_demand_letter_sections_letter_idx
  ON case_demand_letter_sections(letter_id);

-- RLS pattern matches migrations 0055-0058 (org_id check via case_demand_letters join).
```

`cache_hash = sha256(case_id + ':' + claim_type + ':' + demand_amount_cents)`. Recipient/summary/deadline NOT in hash.

## Components — server detail

### `classify.ts`
- Input: `{ caseId, orgId }`. Pulls case row + 10 most-recent case_documents (title + summary) + lawyer-provided `summary` field if available.
- Single Claude **haiku-4-5** structured-output call → 4-class enum + confidence + 1-sentence rationale.
- Throws `CLASSIFY_PARSE_ERROR` on malformed output.
- Unit testable via mocked Claude client.

### `sources.ts`
- `fetchCaseDocsExcerpts(caseId, query, k=5)` — reuse existing `voyage.ts` wrapper (has exponential backoff for 3 RPM). Concurrency=1 within a letter. Empty result valid.
- `fetchStatutesForClaim(claim_type, k=3)` — SQL filter on existing `cached_statutes` table. Tag map:
  - `contract → ['contracts','UCC']`
  - `personal_injury → ['torts','negligence']`
  - `employment → ['labor','wage','wrongful_termination']`
  - `debt → ['fdcpa','collections']`
  - Empty result → fallback to embedding search via Voyage. Empty after that → returns `[]`.
- Both wrapped in `Promise.all`.

### `orchestrator.ts`
- `aiSuggest({ caseId, userId, orgId })` → `{ claim_type, confidence, rationale }`. **No credit charge** (preview).
- `aiGenerate({ caseId, claim_type, demand_amount_cents, deadline_date, recipient_name, recipient_address, summary, letter_type, userId, orgId })`:
  1. Compute `cache_hash`.
  2. SELECT existing letter on `(org_id, cache_hash)`. If found → load sections → return (0 credits).
  3. `decrementCredits(userId, 3)` — atomic, throws `InsufficientCreditsError` (existing util from `services/credits.ts`).
  4. Try block:
     - `Promise.all([fetchCaseDocsExcerpts, fetchStatutesForClaim])`.
     - Sequential `draftSection(ctx, key)` × 5.
     - DB tx: `demandLettersService.createLetter(...)` with `ai_generated=true`, `claim_type`, `claim_type_confidence`, `cache_hash`, `ai_summary` set + insert 5 `case_demand_letter_sections` rows.
     - Unique-violation on `(org_id, cache_hash)` → catch → SELECT existing → return.
  5. On any throw past charge: `refundCredits(userId, 3)` + rethrow.
- `aiRegenerateSection({ letterId, sectionKey, userId, orgId })`:
  - Ownership check (org_id match via `getLetter`).
  - Re-fetch sources (5-min in-memory cache keyed by letterId).
  - `draftSection(ctx, sectionKey)` → UPDATE `case_demand_letter_sections` row + `regenerated_at`.
  - 0 credits.
- `getLetterSections(letterId, orgId)` — ownership check + return ordered sections array.

### `draft.ts`
- `sectionPrompts: Record<SectionKey, (ctx) => string>` — five builders.
- `header`/`facts`/`consequences` claim-type-agnostic; `legal_basis`/`demand` claim-type-specialized.
- Model: Claude **sonnet-4-6** (precision over speed).
- Per letter: 5 calls × ~700 tokens ≈ 3.5k tokens.

### Router additions (`settlement.ts demandLettersRouter`)
- `aiSuggest({ caseId })` → `{ claim_type, confidence, rationale }`
- `aiGenerate({ ... })` → `{ letterId, letterNumber, sections }`
- `aiRegenerateSection({ letterId, sectionKey })` → `{ content_md }`
- `aiGetSections({ letterId })` → `Section[]`

All new procedures gated by middleware `requireBetaOrg('STRATEGY_BETA_ORG_IDS')` (reuse existing pattern from motion-drafter router).

### PDF renderer extension
- `demand-letter-pdf.tsx` currently renders from `letterBody` field.
- Extension: when `ai_generated === true` and sections rows exist, concatenate `header → facts → legal_basis → demand → consequences` markdown → render via existing markdown-to-PDF block.
- Backward compatible: manual letters (ai_generated=false) keep existing rendering path.

## Data flow

1. **aiSuggest** — beta gate → pull case context → Claude haiku → return claim_type. No charge.
2. **aiGenerate** — beta gate + zod (amount > 0 cents and < 1_000_000_000_000 cents (= $10M); deadline 7–90 days from today) → cache hash → cache lookup → charge → `Promise.all([RAG, statutes])` → 5× sequential `draftSection` → tx insert via existing `createLetter` + sections. On throw: refund.
3. **aiRegenerateSection** — ownership check → re-fetch sources (mem-cached) → single `draftSection` call → UPDATE. No charge.
4. **PDF export** — uses existing `demand-letter-pdf.tsx` renderer; new code path reads from sections when `ai_generated=true`.

## Frontend

### `demand-letters-section.tsx` (existing, modified)
- Add second action button "Draft with AI" next to existing "New letter".
- Visible only when `process.env.NEXT_PUBLIC_AI_FEATURES_BETA?.includes(orgId)` OR via tRPC capability check (preferred — server-authoritative).

### `ai-demand-letter-dialog.tsx` (new)
- Modal-based three-step state machine (no separate routes).
- **Step 0 — Classifying:** spinner → call `aiSuggest` → render `{ claim_type, confidence, rationale }`. Confidence ≥ 0.7 → preselect with "Looks right? Continue." Confidence < 0.7 → ranked dropdown forced selection.
- **Step 1 — Form:**
  - `letter_type` (existing enum) — default `pre_litigation`.
  - `recipient_name` (text), `recipient_address` (textarea), `recipient_email` (optional email).
  - `demand_amount_cents` (currency input) — required, validated.
  - `deadline_date` (date picker) — default today+30, validated 7–90 days.
  - `summary` (textarea, ≥ 50 chars) — describes the claim, fed into LLM.
  - Banner if no case documents: "No case documents uploaded — letter quality will be limited."
- **Step 2 — Generated sections:**
  - 5 sections in `<Accordion>` with markdown editor (reuse component from 2.2.3 research memos).
  - Each section: "Regenerate" button (0 credits) + edit-in-place.
  - Top bar: "Save & close" → upserts edits to `case_demand_letter_sections`. "Export PDF" → triggers existing PDF download endpoint.
- After save, dialog closes; section list refreshes; new letter appears with badge "AI" and ai_generated flag.
- Existing manual edit/markSent/recordResponse flows continue to work on AI-generated letters.

## Error handling — summary table

| Failure | Behaviour |
|---------|-----------|
| Voyage rate-limit exhausts retries | Throw → refund → toast "Voyage temporarily unavailable, try again in 1 minute." |
| Claude error mid-section | Tx rolls back → refund 3 credits → toast generic error. |
| Empty case docs | Facts section uses `summary` field; banner warns lawyer. |
| Empty statute results | Legal Basis falls back to general principles; letter usable. |
| Classifier confidence < 0.7 | Frontend forces lawyer dropdown selection. |
| Concurrent double-submit | Frontend disables button; server unique index on `(org_id, cache_hash)` catches duplicate. |
| Non-beta org direct API call | Router middleware throws `FORBIDDEN`. |
| `demand_amount_cents` invalid | zod rejects (front + server). |
| `deadline_date` out of [today+7, today+90] | zod rejects. |
| Wrong-org access | Ownership check throws `NOT_FOUND` (no info leak). |
| Sent letter regen attempt | Block via existing `status==='draft'` guard in `demand-letters-service`. |

## Out of scope (explicit)

- Direct Word/DOCX export (PDF is canonical; users open PDF→Word client-side).
- AI-suggested damages range / breakdown.
- Case-law citations (CourtListener).
- Standalone (no case) generation.
- Counter-demand / defense-side letters (`response_to_demand` letterType still works manually).
- Auto-fill `key_facts`/`legal_basis`/`demand_terms` legacy text fields from sections (we keep them null when ai_generated=true).
- Multi-language demand letters.
- E-sign integration.

## Goal

Plaintiff-side, pre-litigation demand letters generated from an existing case file. Lawyer clicks "Draft demand letter" on a matter page, AI classifies the claim type, RAG injects facts and statutes, Claude drafts five editable sections, lawyer exports DOCX.

## Decisions (from brainstorm)

| # | Question | Choice |
|---|----------|--------|
| 1 | Audience | A — Plaintiff, pre-litigation only |
| 2 | Facts source | A — Case-attached only (RAG over case docs) |
| 3 | Structure | B — 4 templates × per-section AI generation |
| 4 | Damages | A — Lawyer enters demand amount manually |
| 5 | RAG scope | B — Case docs RAG + statute citations |
| 6 | Export | A — DOCX only (PDF/email send deferred) |
| 7 | Pricing | C — 3 credits flat, cache by `case_id + claim_type + demand_amount_cents` |
| 8 | Access | A — Beta-gated via existing `STRATEGY_BETA_ORG_IDS` |
| 9 | Architecture | Approach 1 — copy 4.3 Motion Drafter shape (no shared core yet) |

Deferrals tracked in memory `project_46_future.md`: in-app send flow, AI damages range, case-law citations via CourtListener, standalone (no case) flow, counter-demand letters.

## Claim types (4)

- `contract` — breach of contract
- `personal_injury` — negligence / PI
- `employment` — wage / wrongful termination
- `debt` — debt collection (FDCPA-aware)

Classifier returns one of these four with confidence 0–1. ≥0.7 → auto-pick. <0.7 → lawyer picks from ranked dropdown.

## Architecture

### File layout

```
src/server/services/demand-letter/
  classify.ts        — Claude (haiku) claim_type classifier
  sources.ts         — Voyage RAG over case_documents + statutes filter
  orchestrator.ts    — cache → credit → classify → sources → draft → tx insert
  draft.ts           — 5 per-section Claude (sonnet) prompt builders + caller
  docx.ts            — `docx` lib renderer → Buffer

src/server/routers/demandLetter.ts
  procedures: suggest, generate, regenerateSection, get, list, exportDocx
  router-level middleware: requireBetaOrg('STRATEGY_BETA_ORG_IDS')

src/app/(dashboard)/cases/[caseId]/demand-letters/
  new/page.tsx       — wizard (step 0 preview, step 1 form, step 2 sections + export)
  page.tsx           — list of letters for this case
  [letterId]/page.tsx — single letter editor (re-enters step 2)
```

### Database (migration 0059)

```sql
CREATE TABLE demand_letters (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id uuid NOT NULL REFERENCES organizations(id),
  case_id uuid NOT NULL REFERENCES cases(id),
  claim_type text NOT NULL CHECK (claim_type IN ('contract','personal_injury','employment','debt')),
  claim_type_confidence numeric(3,2) NOT NULL,
  demand_amount_cents bigint NOT NULL CHECK (demand_amount_cents > 0 AND demand_amount_cents < 1000000000000),
  deadline_date date NOT NULL,
  recipient_name text NOT NULL,
  recipient_address text NOT NULL,
  summary text NOT NULL,
  status text NOT NULL DEFAULT 'draft' CHECK (status IN ('draft','finalized')),
  cache_hash text NOT NULL,
  created_by uuid NOT NULL REFERENCES users(id),
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE (org_id, cache_hash)
);
CREATE INDEX ON demand_letters(case_id);

CREATE TABLE demand_letter_sections (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  letter_id uuid NOT NULL REFERENCES demand_letters(id) ON DELETE CASCADE,
  section_key text NOT NULL CHECK (section_key IN ('header','facts','legal_basis','demand','consequences')),
  content_md text NOT NULL,
  regenerated_at timestamptz,
  UNIQUE (letter_id, section_key)
);

-- RLS: org_id match for demand_letters; via letter_id join for sections.
-- Pattern matches migrations 0055-0058.
```

`cache_hash = sha256(case_id + ':' + claim_type + ':' + demand_amount_cents)`. Recipient/summary/deadline NOT in hash — display-only.

## Data flow

1. **suggest** (`POST demandLetter.suggest { caseId }`)
   - Beta gate.
   - Pull matter description + last 10 case_documents titles + summaries.
   - Single Claude haiku call → `{ claim_type, confidence, rationale }`.
   - **No credit charge** (preview-only).

2. **generate** (`POST demandLetter.generate { caseId, claim_type, demand_amount_cents, deadline_date, recipient_name, recipient_address, summary }`)
   - Beta gate + zod validation (amount > 0, < $10M; deadline 7–90 days from today).
   - Compute `cache_hash`. SELECT existing letter; if found, return its sections (0 credits).
   - `credits.charge(orgId, 3)` — atomic, throws `INSUFFICIENT_CREDITS`.
   - Try block:
     - `Promise.all([fetchCaseDocsExcerpts(caseId, query, k=5), fetchStatutesForClaim(claim_type, k=3)])`.
     - Sequential `draftSection(ctx, key)` × 5.
     - DB tx: insert `demand_letters` + 5 `demand_letter_sections`.
     - Unique-violation on cache_hash → catch → SELECT existing → return.
   - On any throw past charge: `credits.refund(orgId, 3, reason)` + rethrow.

3. **regenerateSection** (`POST demandLetter.regenerateSection { letterId, sectionKey }`)
   - Ownership check (org_id match).
   - Fetch sources (5-min in-memory cache keyed by letterId).
   - `draftSection(ctx, sectionKey)` → UPDATE row, set `regenerated_at`.
   - 0 credits.

4. **exportDocx** (`POST demandLetter.exportDocx { letterId }`)
   - Ownership check.
   - `docx.renderDemandLetter(letter, sections) → Buffer` → return base64.

## Components — server detail

### `classify.ts`
- Input: `{ caseId, orgId }`.
- Pulls matter row + 10 most-recent case_documents (title + summary).
- Single Claude haiku-4-5 structured-output call (4-class enum + confidence + rationale).
- Output: `{ claim_type: ClaimType; confidence: number; rationale: string }`.
- Failure modes: malformed structured output → throw `CLASSIFY_PARSE_ERROR` (no fallback — better to fail than mis-classify).

### `sources.ts`
- `fetchCaseDocsExcerpts(caseId, query, k=5)`:
  - Reuses existing `voyage.ts` wrapper (already has exponential backoff for 3 RPM free tier).
  - Concurrency=1 within a letter.
  - Empty result is valid → returns `[]`.
- `fetchStatutesForClaim(claim_type, k=3)`:
  - Tag map: `{contract:['contracts','UCC'], personal_injury:['torts','negligence'], employment:['labor','wage'], debt:['fdcpa','collections']}`.
  - SQL filter on `statutes` table by tag overlap. If empty result, falls back to embedding search (Voyage call 2).
  - Empty result valid → returns `[]`.

### `orchestrator.ts`
- `suggest(input)`, `generate(input)`, `regenerateSection(input)`.
- Exposes a single `runWithRefund(orgId, credits, fn)` helper for charge → try → refund-on-throw symmetry.

### `draft.ts`
- `sectionPrompts: Record<SectionKey, (ctx) => string>` — five builders.
- `header` and `facts` and `consequences` are claim-type-agnostic.
- `legal_basis` and `demand` are claim-type-specialized (different language for contract vs. PI vs. employment vs. debt).
- Model: Claude sonnet-4-6 (precision matters more than speed for legal text).
- Per letter: 5 calls × ~700 tokens ≈ 3.5k tokens.

### `docx.ts`
- Uses existing `docx` dep (already in 2.4.2 motions, 4.5 discovery-response).
- Layout: recipient address block → date → "RE:" line → 5 sections in fixed order → signature block.
- Returns `Buffer`.

### Router middleware
- `requireBetaOrg(envVar)` — checks `process.env[envVar]?.split(',').includes(orgId)`. Reused/extended from existing strategy/motion-drafter wiring.

## Frontend wizard

`/cases/[caseId]/demand-letters/new` — three steps in a single page with state machine.

- **Step 0 — Preview:** loading spinner → call `suggest` → render `{ claim_type, confidence, rationale }`. If confidence ≥ 0.7, show pre-selected claim type with "Looks right? Continue." If < 0.7, show ranked dropdown forced selection.
- **Step 1 — Form:** demand_amount (currency input, cents under the hood), deadline_date (date picker, default +30 days), recipient_name, recipient_address (textarea), summary (textarea, ≥ 50 chars). Validation via zod resolver. If case has no documents, show banner "No case documents uploaded — letter quality will be limited."
- **Step 2 — Generate + edit:** submit → call `generate` → render 5 sections in collapsible `<Accordion>` with markdown editor (existing component from 2.2.3 research memos). Each section has "Regenerate" button. Top bar: "Export DOCX" + "Mark finalized."

`/cases/[caseId]/demand-letters` — list view: table of letters (claim_type chip, demand_amount, deadline, status, created_at, "Open" link).

`/cases/[caseId]/demand-letters/[letterId]` — re-enters Step 2 for an existing letter.

## Error handling — summary table

| Failure | Behaviour |
|---------|-----------|
| Voyage rate-limit exhausts retries | Throw → refund → toast "Voyage temporarily unavailable, try again in 1 minute." |
| Claude error mid-section | Tx rolls back → refund 3 credits → toast generic error. |
| Empty case docs | Facts section uses `summary` field as primary source; banner warns lawyer. |
| Empty statute results | Legal Basis prompt falls back to general principles; letter remains usable. |
| Classifier confidence < 0.7 | Frontend forces lawyer dropdown selection. |
| Concurrent double-submit | Frontend disables button; server unique index on `cache_hash` catches duplicate. |
| Non-beta org direct API call | Router middleware throws `FORBIDDEN`. |
| `demand_amount_cents` ≤ 0 or ≥ $10M | zod rejects at frontend + server. |
| `deadline_date` < today+7d or > today+90d | zod rejects. |
| Wrong-org access | `assertLetterOwnership(letterId, orgId)` throws `NOT_FOUND` (no info leak). |

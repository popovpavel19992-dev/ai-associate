# 2.3.3 Intake Forms / Questionnaires — Design

**Phase:** 2.3.3 (Client Communication → Intake Forms)
**Date:** 2026-04-20
**Status:** Spec — awaiting plan

## 1. Goal

Lawyer builds a structured intake form on a case (title + ordered fields of 8 supported types). Sends to client. Client opens on portal, fills fields with auto-save, submits. Lawyer reviews answers and can download a print-to-PDF copy.

## 2. Non-goals

- Template library. Future 2.3.3b. MVP forms are ad-hoc per case, but schema is stored as JSONB so a template flow can later copy a schema blob into a new form row without migrations.
- Conditional logic (`showIf` branching). Future, if demand surfaces.
- Multi-page wizard navigation. Single-page scrolling form.
- `email` / `phone` / `signature` field types. Validation of email/phone lives in `short_text` via optional zod regex if ever needed.
- Binary PDF generation. Rely on browser print-to-PDF against a print-styled HTML page (no puppeteer, no Fluid-Compute timeout risk).
- Auto-populate case fields from answers.
- "Client started filling" notification to lawyer (spammy; list badge covers it).
- Editing a form's structure after sending.
- Re-opening a submitted form.

## 3. Key Decisions (from brainstorm)

1. **Ad-hoc forms per case**, no template library on MVP. Schema stored as JSONB to make a future template flow a blob-copy.
2. **8 field types:** `short_text | long_text | number | date | select | multi_select | yes_no | file_upload`.
3. **Auto-save drafts.** Each field answer upserts on blur/change (debounced). Lawyer does not see answer content until `submitted` — only a `{answered}/{totalRequired}` count. "Submit" is a distinct lawyer-facing commit.
4. **Required + basic per-type validation only.** No conditional logic. Validation runs both client-side (submit gate) and server-side (authoritative).
5. **Lawyer surface:** new `"intake"` tab on case detail. Portal surface: card on `/portal/cases/[id]` linking to full-screen `/portal/(authenticated)/intake/[formId]`.
6. **Lifecycle:** `draft → sent → in_progress → submitted` (or `cancelled` anywhere pre-submit). Structure **frozen after send**. To change structure: cancel + create new form.
7. **3 notifications:** `intake_form_sent` (portal: email+in_app+push), `intake_form_submitted` (lawyer: email+in_app), `intake_form_cancelled` (portal: email+in_app).
8. **PDF export = print-to-PDF.** Lawyer opens `/cases/[id]/intake/[formId]/print` → browser Save as PDF. Zero dep.

## 4. Data Model

Two new tables. FKs cascade on parent case delete.

### 4.1 `intake_forms`

```
id            uuid PK
case_id       uuid FK → cases.id (cascade)
title         text not null
description   text nullable
schema        jsonb not null                 -- see §4.3
status        text not null check in
              ('draft','sent','in_progress','submitted','cancelled')
              default 'draft'
created_by    uuid FK → users.id (set null)
sent_at       timestamp with time zone nullable
submitted_at  timestamp with time zone nullable
cancelled_at  timestamp with time zone nullable
created_at    timestamp with time zone default now() not null
updated_at    timestamp with time zone default now() not null

index (case_id, status)
index (case_id, created_at desc)
```

### 4.2 `intake_form_answers`

Normalized one-row-per-answer. Allows efficient `answeredCount` aggregation and per-field updates without rewriting the whole form payload.

```
id             uuid PK
form_id        uuid FK → intake_forms.id (cascade)
field_id       text not null              -- matches `id` in schema JSON
value_text     text nullable              -- short_text, long_text, select (stores option value)
value_number   numeric nullable           -- number
value_date     date nullable              -- date
value_bool     boolean nullable           -- yes_no
value_json     jsonb nullable             -- multi_select (array of strings)
document_id    uuid FK → documents.id (restrict) nullable  -- file_upload
updated_at     timestamp with time zone default now() not null

unique (form_id, field_id)
index (form_id)
```

Service enforces "exactly one value column set, matching the field's type."

### 4.3 Form schema (JSONB shape)

```ts
type FieldType =
  | "short_text" | "long_text" | "number" | "date"
  | "select" | "multi_select" | "yes_no" | "file_upload";

interface FieldSpec {
  id: string;                  // stable UUID generated by builder; persists across saves
  type: FieldType;
  label: string;               // required, max 200
  description?: string;        // max 1000
  required: boolean;
  // type-specific:
  options?: Array<{ value: string; label: string }>;  // select | multi_select
  min?: number; max?: number;                          // number
  minDate?: string; maxDate?: string;                  // date (ISO 'YYYY-MM-DD')
}

interface FormSchema {
  fields: FieldSpec[];         // ordered, max 100 fields
}
```

Zod schema validates on updateDraft and rejects shape drift before writing to DB.

### 4.4 Status transitions

- `createDraft` → `draft`. Empty or partial schema allowed.
- `updateDraft` — only allowed when `status === 'draft'`. Otherwise 400 BAD_REQUEST.
- `sendForm` — requires `schema.fields.length >= 1`. Transitions `draft → sent`. Sets `sent_at`. Fires `messaging/intake_form.sent`.
- `saveAnswer` — allowed when `status ∈ {sent, in_progress}`. First save of any answer transitions `sent → in_progress`. Form must own the field_id per current schema.
- `submitForm` — requires all `required` fields answered. Transitions to `submitted`, sets `submitted_at`, fires `messaging/intake_form.submitted`. Idempotent (no-op if already submitted).
- `cancelForm` — allowed in `draft | sent | in_progress`. Transitions to `cancelled`, sets `cancelled_at`, fires `messaging/intake_form.cancelled` (only if form had been sent).

## 5. Backend

### 5.1 Service — `IntakeFormsService`

Location: `src/server/services/intake-forms/service.ts`.

Methods:
- `createDraft({ caseId, title, description?, createdBy })` — inserts row, empty schema `{ fields: [] }`, returns `{ formId }`.
- `updateDraft({ formId, title?, description?, schema? })` — draft-only. Validates schema shape with zod; rejects duplicate `field_id`s and unknown types.
- `sendForm({ formId })` — transitions draft → sent, emits event.
- `cancelForm({ formId, cancelledBy })` — soft cancel, emits event (only if prior status ≠ draft, since drafts were never visible to the client).
- `saveAnswer({ formId, fieldId, value, uploadedByPortalUserId? })` — validates value against field spec; UPSERTs answer row by (form_id, field_id). Returns `{ answerId, status }` (may be the new in_progress status).
- `submitForm({ formId, submitterPortalUserId })` — validates required-fields completeness server-side; transitions to submitted, emits event.
- `listForCase({ caseId, viewerType })` — lawyer sees all; portal filters to `status ∉ {draft, cancelled}`. Each row includes `answeredCount` and `requiredCount` (from schema).
- `getForm({ formId, viewerType })` — lawyer: full schema + answers always; portal: full schema + own-answer values only if status ∉ draft (portal never sees drafts).
- `submittedCount({ userId, orgId })` — sidebar badge helper. Counts submitted forms in accessible cases (mirror `documentRequests.pendingReviewCount` pattern).

Answer-value routing (internal helper `writeAnswerValue`): given a field spec + value, map to the correct column:
- `short_text | long_text | select` → `value_text`
- `number` → `value_number`
- `date` → `value_date` (expects ISO date string)
- `yes_no` → `value_bool`
- `multi_select` → `value_json` (array of selected option values)
- `file_upload` → `document_id`

Any mismatch throws 400 before DB touch.

### 5.2 tRPC routers

**`src/server/trpc/routers/intake-forms.ts`** (lawyer):
- `list({ caseId })`, `get({ formId })`
- `createDraft({ caseId, title, description? })`
- `updateDraft({ formId, title?, description?, schema? })`
- `send({ formId })`, `cancel({ formId })`
- `submittedCount()` (for sidebar)

All protected by `assertCaseAccess(ctx, caseId)` after resolving `form.caseId` where needed.

**`src/server/trpc/routers/portal-intake-forms.ts`** (portal):
- `list({ caseId })`, `get({ formId })` — filters/scopes as per `viewerType='portal'`.
- `saveAnswer({ formId, fieldId, value })` — portal auth verifies the form's case belongs to portalUser's client.
- `submit({ formId })` — portal auth; validates required fields; kicks the `submitted` event.

Portal auth pattern mirrors `portal-document-requests.ts` (inlined `assertPortalCaseAccess`).

### 5.3 Inngest — `intake-form-broadcast`

Location: `src/server/inngest/functions/intake-form-broadcast.ts`. Three functions, one per canonical event, using the same recipient-resolution helpers as 2.3.2 broadcast (copy or lift to shared lib — decide in plan):

- `messaging/intake_form.sent` → `notification.intake_form_sent` per portal user of the case's client.
- `messaging/intake_form.submitted` → `notification.intake_form_submitted` per lawyer recipient (case_members + cases.userId).
- `messaging/intake_form.cancelled` → `notification.intake_form_cancelled` per portal user.

### 5.4 Notification types

Extend `src/lib/notification-types.ts`:

| Type | Recipient | In-app | Email | Push |
|---|---|---|---|---|
| `intake_form_sent` | portal | ✓ | ✓ | ✓ |
| `intake_form_submitted` | lawyer | ✓ | ✓ | ✗ |

Add to `NOTIFICATION_CATEGORIES.cases`. Metadata shapes:

```ts
intake_form_submitted: {
  caseId: string; caseName: string;
  formId: string; formTitle: string;
  recipientUserId: string;

intake_form_cancelled: {
  caseId: string; caseName: string;
```

Handler wiring mirrors 2.3.2 `document-request-notifications.ts` (dedicated consumer functions per type). Portal push gap from 2.3.1/2.3.2 persists here for `intake_form_sent` — flagged, out of scope.

### 5.5 File-upload fields

`file_upload` reuses the portal upload pipeline (`trpc.portalDocuments.upload` → S3 presigned PUT → `confirmUpload`). The portal form UI:

1. User picks file for a `file_upload` field.
2. Component calls `portalDocuments.upload` → gets upload URL + documentId.
3. Uploads to S3.
4. Calls `portalDocuments.confirmUpload` + `portalIntakeForms.saveAnswer({ fieldId, value: { documentId } })`.

Document rows get `case_id` set so the file appears in the case's documents list automatically.

## 6. UI — Lawyer Side

### 6.1 New tab

Add to TABS array in `src/app/(app)/cases/[id]/page.tsx`:
```ts
{ key: "intake", label: "Intake" }
```

### 6.2 Components

Directory: `src/components/cases/intake/`

- `<IntakeTab caseId>` — two-pane layout (list + detail), mirroring `<RequestsTab>`.
- `<IntakeFormsList caseId selectedId onSelect>` — rows with status pill, title, progress (`{answered}/{requiredCount} required` while in-flight; `submitted {relative time ago}` when done), + "New form" button.
- `<IntakeFormDetail formId>` — renders one of four modes based on `form.status`:
  - `draft` → `<FormBuilder>` with inline title/description edit, field list, add/remove/reorder, "Send to client" button.
  - `sent | in_progress` → `<FormProgressView>` — schema read-only, progress count, "Cancel request" button. No answer content.
  - `submitted` → `<FormAnswersView>` — labeled Q&A list + "Download PDF" (opens `/cases/[id]/intake/[formId]/print` in new tab).
  - `cancelled` → muted read-only view.
- `<FormBuilder>` components:
  - Ordered field list with up/down move buttons + Delete.
  - "Add field" dropdown offers the 8 types; new field opens `<FieldEditor>`.
  - `<FieldEditor>` — label, description, required toggle, type-specific subpanel:
    - `select | multi_select` → options list (label+value pairs, min 2 required).
    - `number` → min / max optional.
    - `date` → min / max optional (ISO).
    - Others: no extra.
- `<NewIntakeFormModal>` — just title + optional description. Field editing happens in the builder after creation (avoid a single giant modal).

### 6.3 Sidebar badge

Extend existing composite badge in `src/components/layout/sidebar.tsx`. Add third source:
```ts
const { data: submittedForms } = trpc.intakeForms.submittedCount.useQuery(undefined, { refetchInterval: 30_000 });
const total = (unreadCases?.count ?? 0) + (pendingReview?.count ?? 0) + (submittedForms?.count ?? 0);
```

## 7. UI — Portal Side

### 7.1 Section on case page

`src/app/(portal)/portal/(authenticated)/cases/[id]/page.tsx` — mount `<IntakeFormsCard caseId>` beside `<DocumentRequestsSection>` (stacked vertically; intake above requests).

- Each row: title, progress bar `{answered}/{totalRequired}`, `[Continue]` or `[Start]` button linking to `/portal/intake/[formId]`.
- Collapsed "History" section for `submitted` + `cancelled`.

### 7.3 Full-screen fill page

`src/app/(portal)/portal/(authenticated)/intake/[formId]/page.tsx`:

- Header: form title + description + sticky progress bar.
- Scroll body: field-by-field rendered components per type.
- Sticky footer: save status ("Saved 2s ago" / "Saving…"), Submit button (disabled while required fields missing).

Components in `src/components/portal/intake/`:
- `<IntakePage formId>` — orchestrator; queries `get`, owns local draft state, debounced auto-save.
- Field components `<ShortTextField>`, `<LongTextField>`, `<NumberField>`, `<DateField>`, `<SelectField>`, `<MultiSelectField>`, `<YesNoField>`, `<FileUploadField>`.
- `<SaveIndicator>` — shows "Saving…" during in-flight, "Saved" on success, red error banner on fail with retry.
- Auto-save debounce: 800ms from last change; also flush on blur and before unload.
- Submit: client-side validation first; errors surface inline; then server mutation; on success, redirect to `/portal/cases/[caseId]` with toast.

## 8. PDF export

Route: `src/app/(app)/cases/[id]/intake/[formId]/print/page.tsx`.

Server component; uses `@clerk/nextjs` auth + `assertCaseAccess` equivalent; fetches form + answers; renders:

```html
<article class="print-intake">
  <h1>{form.title}</h1>
  <p class="meta">Case: {case.name} · Submitted: {submittedAt}</p>
  {form.schema.fields.map(f => (
    <section>
      <h3>{f.label}</h3>
      {formatAnswer(f, answers.get(f.id))}
    </section>
  ))}
</article>
```

Print-optimized CSS (`@media print`): no navigation, grayscale palette, 11pt serif, page-break-inside: avoid on each section. Lawyer clicks "Download PDF" button in `<FormAnswersView>` → route opens in new tab → Cmd+P → Save as PDF.

File-upload answers render as "Attached: {filename}" (no inline binary embed; filename is sufficient for a paper file).

## 9. Acceptance Criteria (Manual UAT)

1. Lawyer opens case → Intake tab → "New form" → enters title "Initial Intake" → form created in `draft`.
2. Lawyer adds 5 fields covering each type category (e.g., short_text "Full name", long_text "Describe incident", date "Date of incident", select "State" with options, file_upload "Driver's license"). At least 2 fields `required`.
3. Lawyer clicks "Send to client" → status → `sent`. Portal user receives email + in-app notification.
4. Portal user opens case → sees intake card "Initial Intake — 0/2 required". Clicks Continue.
5. Full-screen page renders all 5 fields in order.
6. Portal user fills 2 fields, waits 1s → "Saved" indicator appears. Closes tab.
7. Lawyer sees intake list row shows `2/2 required` progress (correct count), status still `in_progress`.
8. Portal user reopens → answers persist.
9. Portal user attempts Submit with one required field empty → client-side validation blocks with inline error.
10. Portal user completes all required → Submit → redirected back to case page with toast; form status → `submitted`. Lawyer receives email + in-app.
11. Lawyer opens the submitted form → sees all answers in Q&A layout.
12. Lawyer clicks "Download PDF" → new tab opens print-styled page; `Cmd+P` produces clean PDF.
13. Lawyer creates another draft, deletes it via "Cancel" → status → `cancelled`. Portal does NOT receive notification (form was still in `draft` at cancel; portal never knew about it).
14. Lawyer sends a new form then cancels after send → portal receives `intake_form_cancelled` email + in-app; card disappears from active portal list.
15. Sidebar badge increments by 1 when a new form is submitted; tooltip breakdown shows the increment.
16. File-upload field: portal user picks a PDF → upload succeeds → filename chip appears → answer persisted → lawyer sees "Attached: filename.pdf" on submitted view.

## 10. Testing

**Unit (service layer):**
- Status transitions: draft → sent, sent → in_progress on first answer, in_progress → submitted, draft/sent/in_progress → cancelled.
- `writeAnswerValue` correctness across 8 field types (positive and mismatch cases).
- Required-fields validator: passes when all required answered, rejects when one missing.
- Schema validation: rejects duplicate field_ids, unknown types, select without options, number with min > max.

**Integration (tRPC):**
- Full create → send → answer → submit → view → print-route status.
- Cancel after send fires event; cancel while draft does not.
- Portal cannot see draft forms.
- Lawyer cannot see answer values while form is still in_progress (list shows count only).

**E2E smoke (Playwright):**
- `/cases/[id]?tab=intake` returns <500.
- `/portal/cases/[id]` still returns <500 with new card mounted.
- `/portal/intake/[formId]` (with placeholder) returns <500.

Target: ~15 new tests on top of current 529 baseline.

- Shared recipient-resolution helpers: 2.3.2 broadcast has `portalRecipients` / `lawyerRecipients` helpers inlined. Plan should either reuse by import or lift to `src/server/services/messaging/recipients.ts`. Recommendation: lift, since 2.3.3 is third caller.

- Default organization scoping when `cases.orgId` is null (some legacy cases have it null): mirror 2.3.2 `pendingReviewCount` pattern (fallback to `cases.userId = user.id`).

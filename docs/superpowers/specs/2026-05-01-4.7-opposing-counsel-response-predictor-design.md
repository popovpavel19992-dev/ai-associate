# 4.7 — Opposing-Counsel Response Predictor (design)

**Date:** 2026-05-01
**Phase:** 4.7
**Status:** Spec — awaiting user review
**Predecessor patterns:** 4.2 Strategy, 4.3 Motion Drafter, 4.5 Discovery Response Drafter, 4.6 Demand Letter (`src/server/services/{case-strategy,motion-drafter,discovery-response,demand-letter-ai}/`)
**Migration:** 0060
**Beta gate:** existing `STRATEGY_BETA_ORG_IDS`

## Goal

Give lawyers an evidence-backed forecast of how opposing counsel will respond to a specific filing, plus a general posture readout for the attorney across the case. Two surfaces:

1. **Inline prediction** in Motion Drafter, Demand Letter, and Discovery Response drafters — "Predict opposing response" → scorecard + reasoning + per-field confidence + audit trail.
2. **Opposing Counsel tab** on the case page — manual + auto-extracted attorney identity, optional CourtListener enrichment, and a posture readout (aggressiveness, settle range, typical motion mix).

## Decisions (from brainstorm)

| # | Question | Choice |
|---|----------|--------|
| 1 | Prediction kind | **D** — combo: filing-specific reaction + general posture readout |
| 2 | Profile data sources | **D** — case docs (always) + CourtListener enrichment (when matched). Cross-case org history deferred. |
| 3 | Output format | **C** — scorecard + expandable reasoning |
| 4 | Trigger surface | **C** — inline buttons in drafters + Opposing Counsel tab on case page |
| 5 | Identity capture | **C** — manual entry (reuse `case_parties`) + auto-extract from signature blocks |
| 6 | Pricing | **A** — 2cr per filing prediction, 2cr per posture readout, both cached |
| 7 | Confidence framing | **C** — per-field confidence + low/high ranges (no point estimates) + sources audit trail |
| 8 | Scope guard | v1 strict — see Out-of-Scope below |

## Out-of-Scope (v1)

- Cross-case org-level counsel history
- Email/motion auto-monitoring (re-predict on opposing reply)
- "Predicted vs actual" learning loop
- Multi-counsel teams (lead attorney only)
- Pro se opposing party
- Predictions for non-filing events (settlement conferences, depositions, hearings)
- Standalone module page (lives inside case context)

### Database (migration 0060)

```sql
-- CourtListener mapping for an opposing-counsel case_party
CREATE TABLE opposing_counsel_profiles (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          uuid NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  case_party_id   uuid NOT NULL REFERENCES case_parties(id) ON DELETE CASCADE,
  cl_person_id    text,
  cl_firm_name    text,
  bar_number      text,
  bar_state       text,
  match_confidence numeric(3,2),
  enrichment_json jsonb,
  enrichment_fetched_at timestamptz,
  created_at      timestamptz NOT NULL DEFAULT now(),
  updated_at      timestamptz NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX ocp_org_party_uq ON opposing_counsel_profiles(org_id, case_party_id);
CREATE INDEX ocp_cl_person_idx ON opposing_counsel_profiles(org_id, cl_person_id)
  WHERE cl_person_id IS NOT NULL;

-- General posture readouts (counsel-level, scoped to a case)
CREATE TABLE opposing_counsel_postures (
  id                uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id            uuid NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  case_id           uuid NOT NULL REFERENCES cases(id) ON DELETE CASCADE,
  profile_id        uuid NOT NULL REFERENCES opposing_counsel_profiles(id) ON DELETE CASCADE,
  cache_hash        text NOT NULL,
  aggressiveness    int,
  settle_likelihood numeric(3,2),
  settle_low        numeric(3,2),
  settle_high       numeric(3,2),
  typical_motions   jsonb,
  reasoning_md      text NOT NULL,
  sources_json      jsonb NOT NULL,
  confidence_overall text CHECK (confidence_overall IN ('low','med','high')),
  created_at        timestamptz NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX ocp_posture_cache_uq ON opposing_counsel_postures(org_id, cache_hash);

-- Filing-specific predictions
CREATE TABLE opposing_counsel_predictions (
  id                uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id            uuid NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  case_id           uuid NOT NULL REFERENCES cases(id) ON DELETE CASCADE,
  profile_id        uuid REFERENCES opposing_counsel_profiles(id) ON DELETE SET NULL,
  target_kind       text NOT NULL CHECK (target_kind IN ('motion','demand_letter','discovery_set')),
  target_id         uuid NOT NULL,
  cache_hash        text NOT NULL,
  likely_response   text NOT NULL,
  key_objections    jsonb NOT NULL,
  settle_prob_low   numeric(3,2),
  settle_prob_high  numeric(3,2),
  est_response_days_low  int,
  est_response_days_high int,
  aggressiveness    int,
  recommended_prep  jsonb,
  reasoning_md      text NOT NULL,
  sources_json      jsonb NOT NULL,
  confidence_overall text CHECK (confidence_overall IN ('low','med','high')),
  created_at        timestamptz NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX ocp_pred_cache_uq ON opposing_counsel_predictions(org_id, cache_hash);
CREATE INDEX ocp_pred_case_target_idx ON opposing_counsel_predictions(case_id, target_kind, target_id);
```

Notes:
- **Profile = (org, case_party_id)** — identity scoped through existing `case_parties` row with `role='opposing_counsel'`. CL match optional.
- **`target_id` polymorphic without FK** — Phase 4 pattern (motion_drafts, discovery_responses, demand_letters live in different tables).
- **Ranges stored as low/high** — matches confidence framing decision.
- Cache hash composition (Phase 4 convention): `sha256(...)` of stable identifying fields, scoped per-org by UNIQUE.

### Cache hash composition

```
posture.cache_hash    = sha256(case_id + profile_id + case_state_hash)
predict.cache_hash    = sha256(case_id + target_kind + target_id +
                               target_content_hash + profile_id)
case_state_hash       = sha256(case_id + max(case_documents.created_at) + parties_rev)
target_content_hash   = sha256(stable JSON of target draft body + key fields)
```

7-day TTL via `created_at` predicate at read time (no background sweep).

## Data flow

### Flow A — Inline filing prediction

1. tRPC `opposingCounsel.predictResponse({ caseId, targetKind, targetId })`
2. Compute `target_content_hash`
3. Resolve profile from `case_parties.role='opposing_counsel'` for `caseId`:
   - 0 → `{ needsAttorney: true }`, no charge
   - 1 → use it
   - >1 → `{ needsAttorneyChoice: [...] }`, no charge
4. Compute `cache_hash`; SELECT — if hit, return row, no charge
5. `credits.charge(orgId, 2, 'opposing_counsel.predict')`
6. `identify.ensureCLMatch(profile)` — best-effort CL person search
7. `enrich.fetchIfStale(profile)` — refresh CL aggregates if older than 7d
8. `sources.collect(caseId, targetId)` — Voyage RAG, opposing-authored docs only
9. `predict.run({ target, profile, enrichment, sources })` → Claude sonnet, Zod-validated
10. INSERT `opposing_counsel_predictions`, return row

### Flow B — Posture readout

1. tRPC `opposingCounsel.getPosture({ caseId, profileId })`
2. Compute `case_state_hash`, `cache_hash`
3. Cache lookup → hit returns row, no charge
4. `credits.charge(orgId, 2, 'opposing_counsel.posture')`
5. `identify.ensureCLMatch` + `enrich.fetchIfStale`
6. `sources.collectPostureContext(caseId, profileId)` — all opposing-authored items in this case + key case-state docs
7. `posture.run({ profile, enrichment, sources })` → Claude sonnet
8. INSERT `opposing_counsel_postures`, return row

### Flow C — Auto-extract on document upload

After existing OCR pipeline, when `doc_type ∈ ('motion','letter','correspondence')` and authorship is opposing:

1. `extract.signatureBlock(text)` — Claude haiku, no credit
2. If confidence ≥ 0.7 → emit suggestion event; UI shows toast "Detected attorney X at firm Y. Add to case parties?"
3. User accepts → create `case_parties` row with `role='opposing_counsel'`
4. Lazy: `opposing_counsel_profiles` row created on first prediction request

### Flow D — CourtListener identification

```
identify.ensureCLMatch(profile):
  if profile.cl_person_id is set → return
  if name (+ optional firm/bar) present → CL /api/rest/v4/people/?name=…
  best result above 0.6 confidence → save cl_person_id + match_confidence
  else → leave null (downstream notes "no public history available")
```

## UI surface

### Inline button (drafters)

`[Predict opposing response]` — secondary button next to Generate/Export, disabled until draft saved. Opens `<PredictResponseDialog />`:

- **needsAttorney** — link to tab + inline manual add
- **needsAttorneyChoice** — dropdown of opposing counsel rows
- **loading** — skeleton scorecard
- **result** — scorecard with `●●●` / `●●○` / `●○○` confidence indicators per field; expandable Reasoning + Sources panels; Regenerate button (re-charges)

Scorecard fields: likely response, settle probability range, response timeline range, aggressiveness 1–10, key objections (bullets), recommended prep (bullets). Each carries its own confidence indicator.

### Opposing Counsel tab

New tab on case page next to existing tabs:

- Attorney list (from `case_parties`) with CL match badge, bar number, "Posture" button per attorney
- `[+ Add attorney]` opens manual entry dialog
- Posture card under selected attorney: aggressiveness, settle posture range, typical motions ranked, expandable reasoning + sources, link to "Predictions issued for this counsel (N)"

### Auto-extract toast

`[ⓘ Detected attorney signature: Jane Smith · Smith & Co]  [Add to case parties]  [Dismiss]`

### Beta gate

Hidden when `org_id ∉ STRATEGY_BETA_ORG_IDS`. Server-side: tRPC middleware throws FORBIDDEN.

## Error handling

| Path | Behaviour |
|---|---|
| No opposing counsel | Short-circuit before charge, return `needsAttorney` |
| Multiple counsel, none picked | `needsAttorneyChoice`, no charge |
| CourtListener fails / timeout | Soft-fail: proceed without enrichment, cap `confidence_overall` at `med`, note in sources |
| Voyage returns 0 chunks | Proceed with profile + enrichment only, downgrade confidence to `low` |
| Claude error (5xx / refusal) | Refund 2cr, error toast, no row inserted |
| Claude invalid JSON | Zod parse fail → refund + Sentry + retry toast |
| Cache hash collision | UNIQUE catches, return existing row |
| Auto-extract miss / low confidence | Silent skip |
| Beta-gate bypass | tRPC middleware FORBIDDEN |
| Stale enrichment + CL slow | Use stale, flag in sources |
| Profile deleted mid-prediction | `ON DELETE SET NULL` on predictions; posture cascades |

**Refund rule** (Phase 4 pattern): charge → LLM call → on throw, `credits.refund(orgId, 2, eventId)` in catch.

# 4.9 — Deposition Anticipated-Answer Branches (design)

**Date:** 2026-05-02
**Phase:** 4.9
**Status:** Spec — awaiting user review
**Predecessor patterns:** 4.5 Discovery Response Drafter, 4.7 Opposing Counsel Predictor, 4.8 Settlement Coach
**Migration:** 0063
**Beta gate:** existing `STRATEGY_BETA_ORG_IDS`

## Goal

AI layer over the existing `deposition-prep` infrastructure: for each question in a deposition topic, generate 2–4 anticipated answer branches (admit / deny / evade / idk) with likelihood, likely-response narrative, and 2–3 follow-up questions per branch. Lawyer prep tool — turns a flat outline into a decision tree they can rehearse and refer to during depo.

This is **not** a new outline generator (existing `ai-generate.ts` already produces per-topic questions). It's a second AI step that adds anticipated-answer trees on top.

| 6 | Pricing | **A** — flat 2 credits per topic, 7-day cache, regenerate full price |
| 7 | UI | **A** — inline expandable section under each topic in existing outline view |
| 8 | Scope guard | v1 strict — see Out-of-Scope below |

## Out-of-Scope (v1)

- Live deposition simulator (AI plays witness — defer to 4.9d)
- Whole-deposition strategy: arc, ordering, key admissions (4.9b)
- Document/exhibit recommender per question (4.9c)
- Witness statement cross-check for impeachment
- Per-question regenerate (cache miss = regenerate whole topic)
- Auto-refresh branches when questions edited (lawyer regenerates manually; UI shows stale badge)
- Custom answer-type labels (fixed: `admit / deny / evade / idk`)
- Multi-language (English only — match existing depo flow)
- Print/export branches in PDF renderer

### Database (migration 0063)

```sql
CREATE TABLE case_deposition_topic_branches (
  id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id              uuid NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  case_id             uuid NOT NULL REFERENCES cases(id) ON DELETE CASCADE,
  outline_id          uuid NOT NULL
    REFERENCES case_deposition_outlines(id) ON DELETE CASCADE,
  topic_id            uuid NOT NULL
    REFERENCES case_deposition_topics(id) ON DELETE CASCADE,
  cache_hash          text NOT NULL,

  -- Snapshot of the question set this row was generated against.
  -- Used by UI to detect staleness when lawyer edits questions later.
  questions_snapshot  jsonb NOT NULL,
  -- shape: [{ questionId, number, text }]

  branches_json       jsonb NOT NULL,
  -- shape: [
  --   { questionId,
  --     branches: [
  --       { answerType: 'admit'|'deny'|'evade'|'idk',
  --         likelyResponse: string,
  --         likelihood: 'low'|'med'|'high',
  --         followUps: [{ text: string, purpose: string }]
  --       }
  --     ]
  --   }
  -- ]

  reasoning_md        text NOT NULL,
  sources_json        jsonb NOT NULL,
  confidence_overall  text,

  created_at          timestamptz NOT NULL DEFAULT now(),

  CONSTRAINT cdtb_confidence_check
    CHECK (confidence_overall IS NULL OR confidence_overall IN ('low','med','high'))
);
CREATE UNIQUE INDEX cdtb_cache_uq
  ON case_deposition_topic_branches(org_id, cache_hash)
  WHERE cache_hash IS NOT NULL;
CREATE INDEX cdtb_topic_idx
  ON case_deposition_topic_branches(topic_id, created_at DESC);
CREATE INDEX cdtb_outline_idx
  ON case_deposition_topic_branches(outline_id, created_at DESC);
```

Notes:
- Per-topic immutable cache row. Regenerate via salt produces a new row; old preserved as audit trail.
- `outline_id` denormalized for fast `listBranchesForOutline` without join.
- `branches_json` is a single jsonb (always arrays inside) — no row-per-branch overkill.
- Cache hash: `sha256(topic_id + sha256(sorted question texts) + regenerate_salt)`. Topic edits → cache miss.
- 7-day TTL via `created_at` predicate at read time (no background sweep).

## Compute / extraction conventions

- Claude sonnet model literal `"claude-sonnet-4-6"`
- Inline `getAnthropic()` + `messages.create` pattern (canonical: `src/server/services/demand-letter-ai/classify.ts`)
- Zod with `z.preprocess` on confidence/likelihood enums to tolerate `medium → med` (per 4.8 lesson)
- Zod refine: each question's `branches` array length is in `[2, 4]`; answerType set must be subset of `{admit, deny, evade, idk}` with no duplicates within one question
- Errors propagate (no swallow); orchestrator catches and refunds

## Data flow

### Flow A — generateBranchesForTopic

1. tRPC `depositionBranches.generateBranchesForTopic({ caseId, outlineId, topicId, regenerate? })`
2. `assertBetaOrg(orgId)`
3. Load topic + ordered questions. If 0 questions → throw `NoQuestionsError`, no charge.
4. `questions_hash = sha256(sorted([qN.text]))`
5. `cache_hash = sha256(topic_id + questions_hash + regenerate_salt ?? 0)`
6. Cache lookup → hit returns row, no charge
7. `decrementCredits(userId, 2)` → false → `InsufficientCreditsError`
8. Try:
   1. Load outline (deponentName, deponentRole, servingParty), case (summary)
   2. `sources.collectDeponentSources({ caseId, deponentName, deponentRole })` — Voyage RAG
   3. Load latest `opposing_counsel_postures` for caseId (best-effort, null on miss)
   4. `generate.run({ topic, questions, outline, caseSummary, sources, posture })` → Claude sonnet structured JSON
   5. `INSERT case_deposition_topic_branches`
9. Catch wrapping 8.1–8.5: `refundCredits(userId, 2)`, re-throw

### Flow B — getBranches (read)

```
SELECT row WHERE topic_id = ? AND org_id = ?
ORDER BY created_at DESC LIMIT 1
```
Returns row or null.

### Flow C — listBranchesForOutline

For all topic_ids in the outline, batch SELECT the latest branch row per topic. UI renders inline beneath each topic.

## UI surface

### Mount point

Inside the existing deposition outline detail view, beneath each topic's questions list, render `<TopicBranchesSection topicId={...} ... />`.

### Empty state (no branches yet)

```
▸ Anticipated answers (not generated)
  [Generate anticipated answers (2 credits)]
```

### Generated state

Section collapsed by default; expand to show tree:

```
▾ Anticipated answers (3 of 6 questions)        confidence: HIGH
  Generated 2 days ago · [Regenerate (2cr)]
  ⚠ Questions changed since generation         (only if stale)

  Q1. <question text>
  ┌── ●●● Admit (likely 75%) ─────────────────┐
  │ Likely: "<likely response>"               │
  │ Follow-ups:                                │
  │   → <follow-up 1>                         │
  │   → <follow-up 2>                         │
  └────────────────────────────────────────────┘
  ┌── ●●○ Evade (likely 15%) ─────────────────┐ …
  ┌── ●○○ IDK (likely 10%) ───────────────────┐ …

  Q2. …
```

Staleness logic (client-side): compare `row.questions_snapshot[].text` to current question texts (by questionId). Mismatch on any question → show stale badge.

### Components

- `topic-branches-section.tsx` — wrapper, loads `getBranches({ topicId })`, renders empty state OR tree, computes staleness, owns expand/collapse state.
- `question-branch-tree.tsx` — for one `questionId`, renders 2-4 branch cards. Per card: answerType chip, likelihood dots (●●● / ●●○ / ●○○), likely response (italic), follow-ups list.
- `generate-branches-button.tsx` — calls `generateBranchesForTopic` mutation, shows skeleton, disables close while pending.

### Beta gate

Section returns `null` when not beta-enabled (re-use `trpc.opposingCounsel.isBetaEnabled`). Server enforced via `assertBetaOrg` in orchestrator.

## Error handling

| Path | Behaviour |
|---|---|
| 0 questions in topic | Short-circuit `NoQuestionsError`, no charge |
| Voyage returns 0 chunks | Proceed with case summary only, downgrade `confidence_overall='low'` |
| Voyage finite/dim guard fail | Same — log warn, empty sources |
| `VOYAGE_API_KEY` missing | Empty sources, generate from case summary |
| Posture missing | Proceed without (best-effort) |
| Claude error / refusal | Refund 2cr, error toast, no row |
| Claude invalid JSON / Zod fail | Refund + Sentry + toast |
| Claude branch count out of [2..4] per question | Zod refine rejects → refund |
| Claude unknown answerType | Zod enum rejects → refund |
| Claude `medium` for likelihood | `z.preprocess` normalizes to `med` |
| Cache hash collision | UNIQUE catches → return existing row |
| Concurrent generate same hash | Second hits new row via UNIQUE → returns it |
| Stale snapshot | UI shows badge, lawyer regenerates manually (paid) |
| Topic/outline deleted mid-generation | Cascade drops row; orchestrator catches FK → refund |
| Beta-gate bypass via direct tRPC | TRPCError FORBIDDEN |

**Refund rule** (Phase 4 pattern): charge → LLM + sources + insert in `try` → catch refunds and re-throws.
